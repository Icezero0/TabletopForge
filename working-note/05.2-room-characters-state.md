# Step 5 Phase 2：房间角色库与状态层

**目标**：建立 PRD 要求的 **房间角色库**（`room_characters`）与 **CharacterState** 实时层；PL 可在跑团桌面创建主 PC / 附加角色。

**依据**：[`05-character-core.md`](./05-character-core.md)、`docs/09_module_design/character_card.md` §7、`docs/01` §4.2.4。

**前置**：[05.1-token-foundation.md](./05.1-token-foundation.md)（Token 可独立放置；本 Phase 不强制绑定）。

---

## 数据与 API

### Migration `room_characters`

```text
id                   INTEGER PK（或 room_id + character_id 复合 PK）
room_id              FK → rooms.id
character_id         FK → characters.id
kind                 VARCHAR — pc_main | pc_additional | npc
added_by_user_id     FK → users.id
created_at
```

- 同一 `character_id` 可被多个房间引用（后续扩展）；MVP 每房间一行即可。
- 历史 `monster` 已迁移为 `npc`。

### Migration `characters` 扩展

在已有 [`characters`](../backend/app/modules/character/models.py) 表增加：

```text
kind                 VARCHAR — pc_main | pc_additional | npc，默认 pc_main
token_image_asset_id FK → assets.id, nullable
```

- `portrait_asset_id` 保留（角色卡头像）；`token_image_asset_id` 专用于地图 Token 圆形图。

### Migration `character_states`

1:1 与 `characters`。

```text
character_id         PK/FK → characters.id
current_hp           INT, nullable
max_hp               INT, nullable
temp_hp              INT, default 0
armor_class          INT, nullable
conditions           JSON，默认 {}
damage_taken         INT, default 0（Phase 4 怪物伤害记录用）
updated_at
```

- 创建角色时自动 insert 默认 state 行（`max_hp`/`current_hp`/`armor_class` 可从 attributes.derived 初始化，或留空手填）。

### HTTP API

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/rooms/{id}/characters` | 房间角色库列表（含 character 摘要 + state 摘要） |
| POST | `/rooms/{id}/characters` | 创建 character + room_characters + character_states |
| POST | `/rooms/{id}/characters/link` | 将全局库已有 `characters` 关联进 `room_characters`（幂等；须 `owner_id === 当前用户`；PL 仅 `pc_*`，GM 仅 `npc`） |
| GET | `/characters/{id}/state` | 状态层详情（权限过滤） |
| PATCH | `/characters/{id}/state` | 更新 HP/AC/conditions 等 |

**权限**（扩展 [`CharacterService.require_character_accessible`](../backend/app/modules/character/service.py)）：

- PL：读/写 **自己 owner** 的角色稳定层与 state。
- GM：读/写房间内 **任意** 角色（怪物正文编辑 Phase 4 细化）。
- OB：只读摘要（Phase 3 InfoPanel 消费）。

**Schemas**：`RoomCharacterEntryResponse`、`CharacterStateResponse`、创建 DTO：

- **主 PC**：名称、player_name、identity/attributes/flavor 简版或引用全局 6-Tab 结构、可选 `token_image`。
- **附加角色**：名称、简字段（race/class 可选）、RP 大文本（`flavor`/`extras`）、可选 `token_image`。

### 不做

- Token 强制 `linked_character_id`
- 外部 JSON/XML 批量导入
- GM 怪物创建 UI（Phase 4）

## 本 Phase 做

### 后端

- 三份 migration（`room_characters`、`characters` 扩展、`character_states`）
- `modules/rooms/characters/` 或扩展 character 模块：repo / service / schemas
- rooms API 路由 `GET/POST /rooms/{id}/characters`
- characters API 扩展 `GET/PATCH /characters/{id}/state`
- 创建角色时联动 insert state + room_characters

### 前端

| 项 | 说明 |
|---|---|
| **API client** | `roomCharacters.api.ts`；`characterState.api.ts` |
| **Composable** | `useRoomCharacters(roomId)` — 房间库列表与刷新 |
| **底栏「添加角色」** | `CharacterSpawnPopover`：已入库 + 本人全局库去重合并（`spawnCharacters.ts`）；未入库一键 link 上场；**GM** 见创建者；末尾「+」→ `AddRoomCharacterDialog` |
| **主 PC 表单** | 简化内嵌表单；`player_name` 绑定当前用户显示名（只读）；或跳转 `/characters/new?roomId=&kind=pc_main` |
| **附加角色表单** | 名称 + 简字段 + 大文本 + 可选 Token 图 |
| **GM 快速角色** | 名称、HP、AC、描述 textarea、Token；`kind=npc` |
| **全局角色卡** | 已有 [`CharacterEditPage`](../frontend/src/pages/characters/CharacterEditPage.vue) `returnTo` 接回房间 |
| **房间库列表** | 底栏或临时面板展示已添加角色（完整 InGameCharacterList 留 Phase 3） |

---

## 本 Phase 不做

- Token 与 character 强制绑定
- InfoPanel、InGameCharacterList 业务数据
- GM 怪物表单与伤害记录 UI
- HP/AC/PP Token 预览条
- 属性/技能自动计算引擎
