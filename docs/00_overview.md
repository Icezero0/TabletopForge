# 2 产品概述

TabletopForge 是一个面向 TRPG / DND 跑团场景的在线协作桌面系统。

系统以“房间”为核心协作空间。用户可以创建房间、加入房间，并在房间内完成角色卡管理、地图布置、Token 操作、普通聊天、角色扮演消息、骰子判定、操作日志记录与实时状态同步。

系统优先支持 DND5E 场景，但整体设计应保留规则集扩展空间，后续可支持其他 TRPG 规则或自定义规则。

---

# 3 产品目标

TabletopForge 的核心目标是：

1. 替代分散的角色卡文档与聊天记录，提供统一的跑团工作台。
2. 支持多人在同一房间中进行地图、角色、消息和状态协作。
3. 通过角色卡与状态追踪降低 HP、资源、法术位、状态效果等内容的维护成本。
4. 通过地图桌面与 Token 系统提供轻量在线战斗场景支持。
5. 通过 RP 消息与操作日志沉淀可回溯的跑团记录。
6. 通过 WebSocket 保证房间内关键状态实时同步。

# 5 系统总体架构

系统采用前后端分离架构。

```text
Frontend
  ├─ HTTP API       -> Backend
  └─ WebSocket      -> Realtime Service

Backend
  ├─ Auth / Users
  ├─ Rooms / Members / Permissions
  ├─ Messages / RP Messages
  ├─ Characters / Character States
  ├─ Scenes / Maps / Tokens / Assets
  ├─ Dice / Logs
  └─ Realtime
```

后端采用模块化单体架构。前端通过 HTTP API 进行持久化业务操作，通过 WebSocket 接收房间内实时事件。

| 实时同步模块 | 广播房间内事件并维护在线状态 |

# 7 身份模型总览

系统采用三层身份模型：

```text
site_role：站点级身份
room_role：房间级身份
game_role：游戏级身份
```

三者互相独立，不做隐式绑定。

- `site_role` 用于站点级管理，例如反馈查看、站点角色设置。
- `room_role` 用于房间治理，例如入房审批、成员管理、房间信息维护。
- `game_role` 用于游戏内操作，例如切换地图、移动 Token、修改角色状态、暗骰。

当前后端地基已实现：

- `site_role = user | admin`
- `room_role = owner | manager | member`

`game_role` 后续随游戏业务模块实现。

具体权限矩阵在权限设计文档中维护。

# 8 房间页面总览

房间页面是系统主要业务页面。

房间页面分为两个核心区域：

1. Table 区域  
   展示地图、网格、Token、标记与场景内容。

2. 工作区  
   承载普通聊天、RP 消息、角色列表、资源库、骰子、日志、战斗辅助等功能。

RoomPage

# 9 通信原则

系统存在两类通信方式：

1. HTTP API  
   用于需要持久化、权限校验和明确响应结果的操作。

2. WebSocket  
   用于房间内实时事件同步。

原则：

- HTTP API 负责权威状态写入。
- WebSocket 负责广播服务端确认后的结果事件。
- 前端可以进行乐观更新，但最终状态应以后端返回或广播为准。
- 关键状态变化必须写入操作日志。

# 11 系统设计原则

1. 以房间为核心组织系统结构。
2. 以服务端数据作为权威状态。
3. HTTP 负责持久化业务操作，WebSocket 负责实时同步。
4. 权限设计分层，避免站点、房间、游戏权限混用。
5. 业务模块边界清晰，避免将规则、地图、聊天、日志逻辑混杂。
6. 角色状态变化必须可追溯。
7. 初始版本优先解决实际跑团痛点，避免过早实现重型 VTT 功能。
8. DND5E 优先落地，但保留规则集扩展能力。
